# Style Guide

Best practices for building and maintaining Jekyll + Bootstrap sites. Derived from patterns across the site portfolio (minima starter, hattmgt, omgtennis, sartorius, cordovadoodles, vitalmedts).

---

## Tech Stack

Every site is a **Jekyll** static site styled with **Bootstrap 5** via a **Bootswatch** theme. The shared stack:

- Jekyll (static site generator)
- Bootstrap 5 (CSS framework, loaded via SCSS)
- Bootswatch (theme layer on top of Bootstrap)
- Bootstrap Icons (icon font, loaded via CDN)
- Google Fonts (per-site typefaces, loaded via `@import url(...)`)

### SCSS Import Order

The main stylesheet lives at `assets/style.scss` (or `assets/css/main.scss` in non-minima sites like sartorius). It **must** follow this order:

```scss
---
---

// 1. Google Fonts
@import url('https://fonts.googleapis.com/css2?family=...');

// 2. Optional: override Bootswatch variables BEFORE import
$primary: #212529;

// 3. Bootswatch variables → Bootstrap → Bootswatch overrides
@import
  "variables",
  "bootstrap/bootstrap",
  "bootswatch";

// 4. Custom font variables (AFTER import, since they reference $font-family-sans-serif)
$font-family-heading: 'Poppins', $font-family-sans-serif;

// 5. Base resets (html, body)
// 6. Component styles
// 7. Utility classes
// 8. Responsive overrides
// 9. Animations
```

The `_sass/_variables.scss` file is the Bootswatch theme's variable definitions. **Do not edit this file directly** — override individual variables in `style.scss` before the import block.

## CSS Approach

### 1. Bootstrap utility classes (preferred)

Use Bootstrap's built-in classes whenever possible. This keeps HTML self-documenting and avoids style drift.

```html
<!-- Good -->
<p class="fs-5 fw-bold text-dark mb-3">...</p>

<!-- Avoid -->
<p style="font-size: 1.25rem; font-weight: 700; color: #1a1a1a; margin-bottom: 1rem">...</p>
```

Common Bootstrap utilities to reach for first:

- Spacing: `m-*`, `p-*`, `gap-*`
- Typography: `fs-*`, `fw-*`, `text-*`, `lh-*`
- Display/flex: `d-flex`, `align-items-*`, `justify-content-*`
- Sizing: `w-*`, `mw-100`, `min-vh-*`
- Colors: `text-dark`, `text-white`, `bg-dark`, `bg-light`, `text-primary`
- Borders: `border`, `rounded-*`
- Responsive: `col-md-6 col-lg-4`, `d-none d-lg-inline`

### 2. Custom CSS classes (sparingly)

Define custom classes in `style.scss` only when:

- Bootstrap doesn't provide a utility for the pattern
- A combination of utilities is repeated across many elements
- The style is complex enough to warrant a named abstraction (e.g. `.hero-section`, `.service-card`)

Keep custom classes focused and composable. Avoid one-off classes that are only used once — a Bootstrap utility combination is usually better.

### 3. Inline styles (last resort)

Inline styles should only be used when the value is dynamic or cannot be expressed as a class:

```html
<!-- Acceptable: dynamic data rendered as a style -->
<div style="background-image: url('{{ slide.image }}')">...</div>

<!-- Avoid: static values that could be a class -->
<p style="font-size: 1.1rem; max-width: 700px">...</p>
```

## Fonts

Every site defines three font families in `style.scss`:

| Purpose | Variable | Applied to |
|---------|----------|------------|
| Headings | `$font-family-heading` | `h1`–`h6`, `.hfont`, `.font-heading` |
| Body | `$font-family-sans-serif` (or custom `$font-family-body`) | `body`, general text |
| Monospace | `$font-family-mono` | `pre`, `code`, `.font-mono` |

The heading and mono selectors should always be defined:

```scss
h1, h2, h3, h4, h5, h6, .hfont, .font-heading {
  font-family: $font-family-heading;
}

pre, code, .font-mono {
  font-family: $font-family-mono;
}
```

Load fonts via Google Fonts `@import url(...)` at the top of `style.scss`. Only load the weights you actually use.

## Color Palette

Colors are managed through Bootswatch `$variables` and Bootstrap's semantic color system. Each site overrides these in `style.scss` or `_variables.scss`:

- `$primary` — main brand color, used for buttons, links, accents
- `$secondary` — supporting accent
- `$dark` / `$light` — dark/light backgrounds
- `$body-color` — default text color

Additional brand colors (logo-specific, etc.) should be defined as custom utility classes:

```scss
.text-brand-green { color: #1b6b3a !important; }
.brand-accent     { color: #c8a64b; }
```

Never hard-code hex values in HTML. Use Bootstrap color classes (`text-primary`, `bg-dark`) or define a custom utility.

## Layout Architecture

### Base Layout

All minima-based sites share a common `base.html` layout:

```
<!DOCTYPE html>
<html lang="{{ page.lang | default: 'en' }}">
  {%- include head.html -%}
  <body>
    {%- include header.html -%}
    <main class="container-fluid py-4" aria-label="Content">
      <div class="container">{{ content }}</div>
    </main>
    {%- include footer.html -%}
    {%- include sub-footer.html -%}
  </body>
</html>
```

Sites that need full-width sections (hero, dark bands) should provide a `fullwidth` layout that removes the inner `.container` wrapper so pages can manage their own containers per-section.

Non-minima sites (sartorius) use a `default.html` layout but should still follow the same `head → header → main → footer` structure.

### Layout Variants

Standard layout files:

- `base.html` — shell with container-wrapped content
- `fullwidth.html` — shell without inner container (for marketing pages)
- `home.html` — extends base, wraps `{{ content }}`
- `page.html` — extends base, adds page title
- `post.html` — extends base, adds post metadata

### Pages

Pages live in `_pages/` (included via `include: [_pages]` in `_config.yml`).

Front matter conventions:

```yaml
---
permalink: /about/
layout: page
title: About Us
nav_title: About          # Short label for navbar (optional)
description: Page-level meta description for SEO
---
```

The frontpage always uses `permalink: /` and typically uses the `fullwidth` layout.

## Includes (Components)

Reusable HTML fragments live in `_includes/`. The standard set from the minima starter:

| Include | Purpose |
|---------|---------|
| `head.html` | `<head>` tag: meta, stylesheet, SEO, analytics |
| `header.html` | Navbar |
| `footer.html` | Site footer |
| `sub-footer.html` | Below-footer strip (attribution, copyright) |
| `nav-items.html` | Navbar link loop |
| `seo.html` | OpenGraph / meta tags (or `{% seo %}` plugin) |
| `social.html` | Social media icon links |
| `google-analytics.html` | GA snippet |
| `custom-head.html` | Per-site `<head>` additions |
| `carousel.html` | Bootstrap carousel component |

Add new includes for repeated UI patterns (e.g. `hero.html`, `call-to-action.html`, `heading.html`). Name them descriptively and keep them focused on a single concern.

## Data Files

Use `_data/*.yml` files to separate content from markup. This makes content editable without touching HTML.

Good candidates for data files:

- Contact info, addresses, phone numbers (`contact.yml`, `company.yml`)
- Service lists (`services.yml`)
- Social media links (`social.yml`)
- Testimonials, team members, FAQ items
- Client/partner logos

Reference in templates: `{{ site.data.company.phone }}`, `{% for item in site.data.services.items %}`.

## Collections

Use Jekyll collections for content types that have individual pages (projects, puppies, lessons, etc.):

```yaml
# _config.yml
collections:
  projects:
    output: true
```

Each collection item gets its own layout (e.g. `project.html`, `puppy.html`).

Standard fields every site should define:

```yaml
url: "https://example.com"
baseurl: ""
permalink: pretty
title: Site Name

  Site description for SEO.

# Bootswatch theme

  - jekyll-seo-tag

  - _pages

  - .gitignore

  - Gemfile.lock

  - README.md
  - node_modules
  - package.json
  - package-lock.json
  - justfile
  - tests
  - playwright.config.js
  - _design
```

## Sections & Spacing

Structure full-width marketing pages as a sequence of `<section>` elements:

```html
<section class="section py-8">
  <div class="container">
    <h2 class="section-title">...</h2>
    <p class="section-subtitle mx-auto">...</p>
    <!-- content -->
  </div>
</section>
```

Alternate section backgrounds for visual rhythm: default → `section-light` → `section-dark` → etc.

### Extended Spacing Utilities

Bootstrap stops at level 5. Define levels 6–9 for generous section spacing:

```scss
$extra-spacers: (6: 4rem, 7: 5rem, 8: 6.5rem, 9: 9rem);

@each $key, $val in $extra-spacers {
  .py-#{$key} { padding-top: $val !important; padding-bottom: $val !important; }
  .my-#{$key} { margin-top: $val !important; margin-bottom: $val !important; }
  // ... etc
}
```

Reduce these on mobile with a media query override.

### Wrap Utilities

For constraining content width below `.container`:

```scss
.wrap-sm  { max-width: 540px;  margin-left: auto; margin-right: auto; }
.wrap-md  { max-width: 720px;  margin-left: auto; margin-right: auto; }
.wrap-lg  { max-width: 960px;  margin-left: auto; margin-right: auto; }
.wrap-xl  { max-width: 1140px; margin-left: auto; margin-right: auto; }
.wrap-xxl { max-width: 1320px; margin-left: auto; margin-right: auto; }
```

## Animations

Use CSS transitions triggered by an Intersection Observer. The pattern:

1. Add `.animate-on-scroll` plus a direction class (`.animate-fade-up`, `.animate-slide-left`, etc.)
2. Elements start hidden (`opacity: 0; transform: ...`)
3. The observer adds `.is-visible` (or `.animate-in`) when the element enters the viewport
4. Stagger children with `.animate-stagger` and `transition-delay`

Always respect reduced motion:

```scss
@media (prefers-reduced-motion: reduce) {
  .animate-on-scroll {
    opacity: 1 !important;
    transform: none !important;
    transition: none !important;
  }
}
```

The observer script belongs in `base.html` after the footer, not inline on individual pages.

## Images

- Store in `assets/imgs/` (or subdirectories like `assets/imgs/hero/`)
- Use `.webp` format where possible for smaller file sizes
- Always include meaningful `alt` text
- Use `relative_url` filter: `{{ '/assets/imgs/photo.webp' | relative_url }}`
- For hero/background images, use `data-bg` attributes loaded by JS or inline `background-image` styles (acceptable inline style use case)
- Provide an OpenGraph image at 1200×630 (`og-image.jpg` or `opengraph-art.jpg`)

## SEO & Meta

- Use `jekyll-seo-tag` plugin (preferred) or a custom `seo.html` include
- Set `title` and `description` on every page via front matter
- Provide a site-wide `image` in `_config.yml` for social sharing
- Use `jekyll-sitemap` for automatic sitemap generation
- Control indexing with `robots_allow` in `_config.yml` — set to `false` until launch

---

## Accessibility

- Use `aria-label="Content"` on `<main>`
- Ensure color contrast meets WCAG AA (set `$min-contrast-ratio` in variables)
- Provide `alt` text on all `<img>` elements
- Use semantic HTML: `<header>`, `<main>`, `<footer>`, `<nav>`, `<section>`
- Keyboard-navigable components (Bootstrap handles this for dropdowns, modals, carousels)
- **Heading hierarchy**: Heading levels must be sequential — never skip from `h2` to `h4` or `h5`. Use Bootstrap display/size classes (`class="h5"`, `class="fs-4"`, etc.) to control visual size independently of the semantic level. For example, a card title inside an `h2` section should be `<h3 class="h5">`, not `<h5>`.

## Conventions

- **Permalink style**: `pretty` (produces `/about/` not `/about.html`)
- **Date format**: `"%b %d, %Y"`
- **HTML comments**: Use `<!-- Section Name -->` to delineate major page sections
- **Liquid whitespace**: Use `{%- -%}` (trimmed tags) in includes and layouts
- **Feature flags**: Use `_config.yml` booleans (`pin_protect`, `disable_mobile`, `robots_allow`) to control site behavior without code changes
- **Config layering**: Use `_config_live.yml` and `_config_rev.yml` for environment-specific overrides, merged at build time
- **Copyright year**: Always dynamic: `{{ 'now' | date: '%Y' }}`
